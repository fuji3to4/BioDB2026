# 演習2 補足2: データベース設計（応用）

> **対象環境:** Docker Compose (PostgreSQL 17 + PHP/Apache + Next.js 15)
> リポジトリをクローンして `docker compose up` で起動できます。

## データベースは最初の設計が重要！

データベースを構築するためには、最初の設計が重要です。必要なデータやデータベースをどのように使うのかなど、情報を整理したあとテーブルを作成していかないと、うまく機能しないデータベースになってしまいます。
2日目の課題で、各自データベースを作成しますが、「2つ以上のテーブルを外部キーで結合できるようにする」という課題は、テーブルを2つ作ればいいという意味ではありません。この意味がわからないままテーブルを作成してしまい、結合できないテーブルにレコード数をどんどん増やしていってしまって、気づいたときは時すでに遅し・・・ということがあります。そのような結果にならないためにも、最初の設計は、とても重要です。

## ◆データベースを設計しよう

### 1. データベースの目的を考える

当たり前ですが、「何を作るのか」を考えましょう。いいかえれば「何があれば出来上がるのか」を考えるということです。
たとえば住所録ならば、「相手の名前」「住所」「電話番号」があればデータベースが出来上がります。

### 2. 出力フォーマットを決める

集めた情報をどのような形に加工するのか、大まかに考えておきましょう。
データベース演習では、検索サイトを目標としていので、どのキーで検索をするのか、検索結果の表示をどうするか、など、目標とするデザインを考えておけば、これに必要な情報や計算方法が見えてきます。

### 3. 必要な情報を整理する

2.を作るために必要な情報を集めて、一覧表を作ります。

例として、複数のユーザが点数をつけることができる「福岡県グルメ採点＆検索サイト」を作ることを目標とした場合、必要なデータは、「店舗名」「住所」「電話番号」「市町村名」「種別」「点数」があげられます。さらに、市町村名を五十音順でソートしたい場合は「市町村名ふりがな」を加える・・・という感じで、必要なデータをあげていきます。

例として、福岡県グルメ採点＆検索サイトのデータ一覧表を下図のように作成します。

### 4. 正規化する

データベースは１つのテーブルでも動かすことは可能ですが、データが増加すると重複するデータを１つ１つ登録する手間が発生したり、検索に時間がかかるなど、無駄が出てきます。

一覧表から意味のあるグループ別にテーブルを分けたり、計算で求められるような項目は削除するなど、効率的にデータを蓄積できるようにすることを「**正規化**」といいます。

一覧表から、繰り返し出てくる項目を分けます。例えば、「店舗名」「市町村名」「市町村名ふりがな」「種別」は何度も出てきます。一覧表のままでは、市町村名が変更になった場合、すべてのレコードを修正する必要が出て、修正ミスが発生することもあります。表を分けることで、同じデータを何度も入力する必要がなくなります。

また、採点が複数回設定される構成としているため、「店舗名」も繰り返し情報を入力する必要があります。この場合、店舗名が決まれば「住所」「電話番号」「市町村」「種別」が決まってくるため、これらを別のテーブルにわけます。

## ◆結合できないテーブル例（悪い例）

2日目の「2つ以上のテーブルを外部キーで結合できるようにする」という課題を、テーブルを2つ作ればいいという考えで作ったり、グループ分けすればいいという考えで作ると、SQLやPHPで検索文を作る際に苦労することになります。

悪い例：　正規化する前の一覧表を、「市町村別」や「種別」でテーブルを分ける。

**飯塚市**

| 番号 | 店舗名 | 住所 | 電話番号 | 種別 | 点数 |

このような分け方をすると、「外部キー」を使った連結が難しくなり、課題をクリアしたとは言えなくなります。
